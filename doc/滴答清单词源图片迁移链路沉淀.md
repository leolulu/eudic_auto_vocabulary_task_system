# 滴答清单词源图片迁移链路沉淀

本文档沉淀本次在 `背单词` 项目内处理词源截图、识别图片内容、匹配单词任务、并将图片迁移到目标单词正文中的完整有效链路。

目标不是记录所有猜测，而是把这次已经验证过的、确实带来信息增量的做法、结论、失败教训和后续操作规范都沉淀下来。

---

## 1. 任务背景

业务背景：

- `背单词` 是主项目/文件夹。
- 词源信息的“入口方式”目前仍处于试验阶段，不应视为固定方案。
- 本次实际处理的数据入口，是一个父任务叫 `词源收集`，其下有多个空标题子任务，子任务里直接挂截图附件。
- 但未来入口可能变化，例如：
  - 同一个任务里连续插入多张图片，而不是拆成多个子任务。
  - 图片先下载到本地文件夹，再由脚本或 UI 做批量分类导入。
  - 图片和文本混合记录，再由脚本做识别和分流。
  - 不再依赖 `词源收集` 这个父任务名，而改成其他收集容器或本地中转目录。
- 因此，本文档应理解为：
  - 上游“入口/收集”部分保持灵活。
  - 下游“归一化 -> 识别 -> 匹配 -> 上传 -> 回填 -> 完成源任务”这一段是相对稳定的。
- 每张截图表示某个单词的“词源来源/看到这个词的出处”。
- 最终目标是把这些截图“剪切”到对应单词任务的正文中，而不是长期留在某个临时收集入口里。

最终成功的业务动作是：

1. 从某个词源入口收集到图片或图片引用。
2. 读取图片内容。
3. 识别截图对应的单词。
4. 找到 `背单词` 中对应的目标单词任务。
5. 将图片上传到目标任务。
6. 以“正文内嵌图片”的形式插入到目标任务正文中。
7. 原入口中的源项在确认迁移成功后再标记完成、归档或删除。

### 1.2 面向“只看 Markdown 的 agent”的最低执行目标

本文档后续内容默认服务于这样一种 agent：

- **不读取项目源码**
- 只读取本 Markdown 文档
- 但允许使用其所在运行环境已经提供的通用能力，例如：
  - 读取/更新任务正文
  - 下载附件
  - 上传附件
- 代码级任务读写与附件处理能力

因此本文档的目标不是教 agent “理解代码”，而是让它仅凭文档就知道：

1. 当前应先判断哪种入口模式。
2. 每一步该做什么。
3. 什么时候可以自动继续。
4. 什么时候必须停下来，不许乱猜。

如果未来 agent 连上述通用能力都没有，而只有一份 Markdown，则它最多只能做规划和人工辅助判断，不能安全执行落地操作。

### 1.3 执行时默认已具备的能力（禁止提前检查）

只看本文档的 agent，在收到执行指令后**直接进入执行，禁止提前逐项核对以下能力**：

- 调用成功 → 说明能力确实存在，继续
- 调用报错说缺能力/缺权限/缺认证 → 自然停止并报告缺失项即可

以下列出的是最小能力契约，供**报错时定位问题**用，不是供执行前核对用的清单。任一项在实际调用中确认为缺失时，**停止并说明缺什么**，不要自己脑补替代方案：

1. **已具备认证后的 Dida365/TickTick 访问能力**
   - 建议直接使用 API 会话。

2. **能读取目标项目任务列表**
   - 至少能读取 `背单词` 项目下的活跃任务标题、正文和附件信息。

3. **能下载源图片**
   - 如果只能看到任务但无法拿到图片字节，停止。

4. **能上传图片到目标任务**
   - 如果只能识别图片但无法上传，停止。

5. **能更新任务正文**
   - 如果不能回写正文中的 `![image](...)` / 说明文字，停止。

6. **若采用 mode B，必须能够进行代码级附件失活清理**
   - 当前已验证路径是项目内的附件失活代码入口。
   - 如果需要清理附件，而当前环境不能调用这条代码路径，停止。

**再次强调：不要在运行前先花时间去验证上面这些能力是否存在。直接跑，缺了什么自然会在调用时报出来。**

### 1.4 Markdown-only agent 的硬性停止条件

以下情况必须停止，不允许继续猜：

- 入口模式无法判定。
- OCR 文本质量太差，无法形成有效候选。
- 候选第一名和第二名差距过小。
- 当前活跃任务中不存在高置信度对应项。
- 高置信度图片已上传，但目标任务渲染未验证。
- mode B 下正文已改，但源任务附件仍未正确清理。

### 1.1 入口策略说明

这一部分必须明确：

- 当前“`词源收集` 父任务 + 空标题子任务 + 图片附件”的模式，只是**当前实验性入口模式 A**。
- 它不是唯一方案，也不应该在后续沉淀里被写死成唯一前提。
- 后续凡是涉及“查源图”“找源项”“完成源项”的描述，都应理解为：
  - 先识别当前使用的是哪一种入口模式。
  - 再将其归一化为统一的“待处理图片源集合”。

推荐把入口方式抽象成以下几类：

- 模式 A：父任务 + 多个空标题子任务 + 图片附件。
- 模式 B：单个任务内放多张图片，脚本后续再拆分。
- 模式 C：本地文件夹收集图片，再批量导入。
- 模式 D：图片与文字混合记录，再做自动分类。

其中模式 B 在本次已经得到真实验证：

- 源任务示例：`词源source截图`
- 任务本身没有子任务，而是在同一个正文中连续放多张 `![image](...)`
- 处理时不能把整个任务当成一个整体判断；必须先拆成“单张图片 work item”再继续

无论入口如何变化，只要最终能归一化出：

- 源项 ID（或本地文件标识）
- 图片附件/图片文件
- 必要的上下文元数据

后续处理都可以复用同一条下游链路。

### 1.5 入口模式判定顺序（Markdown-only 决策树）

当 agent 只看本文档时，应按下面的顺序判断当前入口模式；一旦命中，就按该模式处理，不要同时混用多个模式的规则：

1. **模式 B：单任务多图**
   - 条件：存在一个源任务，正文里连续出现多张 `![image](...)`，且这些图片都挂在同一个任务下。
   - 当前典型例子：`词源source截图`

2. **模式 A：父任务 + 多个空标题子任务**
   - 条件：存在一个收集父任务（如 `词源收集`），其下有多个空标题子任务，每个子任务内有图片附件。

3. **模式 C：本地文件夹导入**
   - 条件：用户明确指定了本地图片目录，而不是滴答中的任务容器。

4. **模式 D：图文混合记录**
   - 条件：源任务或源容器中同时混有图片和说明文本，需要先抽取出图片 work item。

5. **都不命中时**
   - 输出：`当前源项不符合已知入口模式，停止并请求人工确认，不要自行发明新规则。`

## 2. 本仓库相关代码入口

> 这一节的定位不是“要求未来 agent 去读源码”，而是提供**源码能力索引**。
> 未来 agent 正确的行为应该是：
> - 先看本文档
> - 再按本文档指定的入口去调用现有能力
> - 而不是先自己在仓库里搜索、猜测、理解整套源码

本次链路主要依赖这些文件：

- `config.yaml`
  - 保存 `dida365_username` / `dida365_password`。
- `constants/dida365.py`
  - 包含 `VOCAB_BOOK_PROJECT_ID`，即 `背单词` 项目 ID。
- `dida365_project/api/dida365.py`
  - Dida365/TickTick API 封装。
  - 负责登录、同步、批量更新任务、上传附件等。
- `agent/dida365.py`
  - 更高层的任务处理逻辑。
  - 原本用于“上传词典音频并重排正文”。
- `dida365_project/models/task.py`
  - `Task` 模型。
  - 包含 `status`、`content`、`attachments`、`parentId` 等字段。
- `dida365_project/models/attachment.py`
  - `Attachment` 模型。
  - 包含 `id`、`fileName`、`fileType`、`path` 等。
- `dida365_project/models/upload_attachment.py`
  - 上传附件时的包装对象。
  - 原仓库里默认围绕 `![file](...)` 处理附件引用。

这几处文件足够支撑：

- 查任务
- 查子任务
- 查附件
- 下载附件
- 上传附件
- 更新正文
- 完成原任务

### 2.1 源码能力索引（看文档就知道该调哪里）

未来 agent 如果需要借用项目里的现成能力，应优先按下面的索引使用，而不是自己全局搜索：

| 目标 | 优先入口 | 作用 |
| --- | --- | --- |
| 获取已登录 API 会话、同步项目与任务 | `dida365_project/api/dida365.py` → `Dida365` | 登录、`get_latest_data()`、`post_task()`、`upload_attachment()` |
| 查找某个任务 | `agent/dida365.py` → `Dida365Agent.find_task()` | 按标题取任务 |
| 搜索候选任务 | `agent/dida365.py` → `Dida365Agent.search()` | 用关键词在任务池里搜索 |
| 更新任务正文 | `dida365_project/models/task.py` → `Task.update_content()` + `Task.gen_update_data_payload()` | 回写正文 |
| 更改任务状态 | `dida365_project/models/task.py` → `Task.change_status()` + `Task.gen_update_data_payload()` | 完成/恢复任务 |
| 上传图片到目标任务 | `dida365_project/models/task.py` → `add_upload_attachment_post_payload_by_bytes()` + `dida365_project/api/dida365.py` → `upload_attachment()` | 上传附件 |
| 下载源图片 | 不需要源码推理，直接使用本文档给出的已验证下载公式 | 下载图片字节 |
| mode B 附件级清理 | **优先项目代码的附件失活路径** | 当前最稳的已验证方式 |

### 2.2 文档驱动执行原则（禁止先做源码考古）

未来 agent 必须遵守以下规则：

1. **第一优先级：禁止调查，直接按文档运行**
   - 这个项目采用的是**快速进行、快速失败**的执行方式。
   - 当文档已经给出稳定入口、调用顺序和完成标准时，默认这些入口就是可用的、当前现场状态也与文档描述一致。
   - **禁止**在真正执行前额外花时间确认入口脚本是否存在、实现是否齐全、源任务是否“真的”处于文档描述状态。
   - 正确默认是：**先运行**。
   - 如果运行失败，再根据实际报错、返回结果或用户反馈进入下一步判断。

2. **先按文档选入口，再调用源码能力**
   - 先判断 mode A / mode B / mode C / mode D
   - 再调用对应能力

3. **禁止先全局 grep 仓库再决定怎么做**
   - 这是低效且高风险的
   - 尤其在删除附件、图片引用、上传逻辑这些地方，容易再次踩到已知坑
   - 除非已经发生真实执行失败，否则禁止把“调查仓库”当成默认第一步

4. **优先使用文档中已验证过的能力组合**
   - 例如：
      - 下载图 → 用已验证下载公式
      - 上传图 → 用 `upload_attachment()` 这条路径
      - 更新正文 → 用 `Task.update_content()` + `post_task()`
- mode B 清理 → 用项目代码的附件失活入口

5. **只有当文档明确写着“当前未验证”时，才允许进入探索模式**
   - 例如某个 API 删除 payload 当前不可靠
   - 这时不能当默认方案直接用

6. **如果文档与源码表现冲突，以文档中“已验证行为”优先**
   - 因为文档记录的是已经在真实任务里跑通过的路径
   - 某些源码能力虽然存在，但未必代表在当前业务场景下可靠

7. **失败之后再调查，禁止调查之后再运行**
   - 允许触发调查的典型情况只有：
     - 实际调用报错
     - 返回结果与文档已验证行为冲突
     - 用户反馈说客户端表现与预期不一致
   - 在这些情况出现前，默认策略都应是直接执行，而不是提前做风险研究

8. **默认自主判断并继续执行，只有真正卡住时才向用户请示**
   - 本项目的默认策略仍然是：按文档直接运行，尽量不要打断用户。
   - 以下情况**不应**默认触发请示：
     - 入口脚本是否存在的前置核对
     - 运行环境能力是否存在的前置核对
     - 轻微的文档/现场差异，但仍可按文档主链路继续执行
   - 只有在下面这些情况出现时，才应向用户请示：
     - 文档已经无法继续指导下一步
     - 实际失败现象与文档主链路明显冲突，继续自主处理会增加风险
     - 再继续尝试只会浪费时间，而不是带来有效推进
   - 也就是说：**优先自己判断、自己执行；只有真正卡住时再请示用户。**

### 2.3 推荐调用顺序（先文档，后源码）

在绝大多数执行场景下，推荐顺序固定为：

1. 读本文档，确定入口模式
2. **禁止额外调查，直接进入执行**
3. 根据本文档决定处理单位（单张图片 work item）
4. 借用 `Dida365` 读取任务和附件
5. 用文档给出的下载公式下载图片
6. 做 OCR 和匹配判断
7. 若高置信度，则：
    - 用任务上传入口上传图片
    - 用任务正文更新入口回写 `![image](...)`
8. 若 mode B 且需要清理，则优先走项目代码的附件失活路径
9. 再做最终验收

补充说明：

- 这里的“先读文档”是为了拿到**执行指令**，不是为了进入一轮新的调查。
- 只要用户明确要求执行，就应把文档当成可直接运行的 runbook。
- 除非实际运行失败，否则不需要先确认“脚本在不在”“实现有没有”“状态是不是和文档完全一致”。

这意味着：

- **源码是执行组件**
- **文档才是调度和决策中枢**

## 3. 如何查任务

### 3.1 登录与同步

仓库现成做法是：

```python
import yaml
from dida365_project.api.dida365 import Dida365

cfg = yaml.safe_load(open('config.yaml', encoding='utf-8'))
dida = Dida365(cfg['dida365_username'], cfg['dida365_password'])
```

`Dida365(...)` 初始化后，会走现有登录流程并获取同步数据。

在这次任务里，最常用的是：

```python
dida.get_latest_data()
```

同步完成后可以读取：

- `dida.active_tasks`
- `dida.data['syncTaskBean']['update']`

其中：

- `active_tasks` 更适合直接按 `Task` 模型对象做过滤。
- `syncTaskBean.update` 更适合看原始字段、确认 `childIds` / `sortOrder` / `attachments` 等底层结构。

### 3.2 过滤 `背单词` 项目的有效任务

主过滤条件：

```python
from constants.dida365 import VOCAB_BOOK_PROJECT_ID

tasks = [
    t for t in dida.active_tasks
    if t.project_id == VOCAB_BOOK_PROJECT_ID and t.status == 0
]
```

结论：

- `status == 0` 表示活跃任务。
- `status == 2` 表示已完成任务。

### 3.3 当前实验入口模式 A：找到 `词源收集` 子任务

本节仅适用于本次的实验入口模式 A。

本次数据里，`词源收集` 是一个父任务；目标截图任务都满足：

- `title` 为空字符串或空白。
- `parent_id` 指向 `词源收集` 的任务 ID。

过滤方式：

```python
empty_source_tasks = [
    t for t in dida.active_tasks
    if t.project_id == VOCAB_BOOK_PROJECT_ID
    and t.status == 0
    and not (t.title or '').strip()
    and t.parent_id == SOURCE_PARENT_ID
]
```

本次已确认：在模式 A 下，这些空标题子任务全部都挂在 `词源收集` 下面。

注意：

- 这只是本次输入形态的事实，不是未来的永久规则。
- 后续如果改成模式 B / C / D，这一节应替换为对应入口的“源项发现规则”，而不是继续强行套 `parentId == 词源收集父任务ID`。

### 3.4 当前实验入口模式 B：单任务多图

本次还验证了另一种真实入口：

- 一个任务标题为 `词源source截图`
- 该任务没有子任务
- 正文内连续出现多张 `![image](...)`
- `attachments` 中保存多张图片，且 `createdTime` 可以辅助校验图片顺序

模式 B 的核心不是“一个任务 = 一个待处理单元”，而是：

- 一个任务只是容器
- 每一张图片才是后续 OCR、匹配、迁移的独立处理单元

因此在模式 B 下，推荐先做“归一化拆分”：

1. 按正文中 `![image](...)` 的出现顺序抽取图片引用。
2. 用 `attachments.createdTime` 做顺序校验。
3. 为每张图分配稳定本地编号，例如 `01`、`02`、`03`。
4. 将每张图转成独立 work item，再进入下游流程。

## 4. 如何找图片

这一节默认前提是：上游入口已经被归一化为一组“待处理源项”，并且每个源项都能定位到图片附件或本地图片文件。

### 4.1 先看正文 content，不要先假设有文本

最初要确认一点：当前源项的 `content` 里是否有真正文本。

本次确认结果：

- 在本次模式 A 里，这批空标题子任务的 `content` 主要是图片引用，不是直接可读文本。
- 形态类似：

```markdown
![image](69c63f02.../69c63f02....png)
```

也就是说，真正的信息在附件图片里，不在正文纯文本里。

### 4.2 附件元数据长什么样

附件对象通常包含：

- `id`
- `refId`
- `path`
- `fileName`
- `fileType`
- `createdTime`

其中对后续最关键的是：

- `project_id`
- `task_id`
- `attachment_id`

### 4.3 图片下载的最终有效公式

这一步踩过坑最多。

一开始尝试过：

- 直接请求 `attachment.path`
- 拼 `https://api.dida365.com + path`
- 拼 `https://dida365.com + path`
- 拼若干 CDN host + `path`

这些都不可靠，很多是 `404`。

最终确认的**有效下载 URL 公式**是：

```text
https://api.dida365.com/api/v1/attachment/{project_id}/{task_id}/{attachment_id}
```

加 `?action=download` 也可：

```text
https://api.dida365.com/api/v1/attachment/{project_id}/{task_id}/{attachment_id}?action=download
```

这是本次最关键的突破之一。

示例代码：

```python
url = f'https://api.dida365.com/api/v1/attachment/{task.project_id}/{task.id}/{att.id}?action=download'
resp = dida.session.get(url, headers=dida.headers, timeout=60)
resp.raise_for_status()
image_bytes = resp.content
```

## 5. OCR 与图片识别阶段

### 5.1 纯内存下载没问题

本次已确认：

- 图片下载到内存完全没问题。
- 可以用 `BytesIO` / `Pillow` 做内存内压缩、转码、base64 化。

### 5.2 真正不稳定的是 OCR 工具，不是图片下载

尝试过两条 OCR 路：

1. 纯内存图片 -> OCR
2. 临时文件 -> OCR

结果：

- 纯内存图片输入 OCR 工具时，这次会出现无响应/不稳定。
- 临时文件方式更稳，实际成功识别了样例图片。

所以业务上最稳的方案是：

- 下载图片到系统临时目录
- OCR 识别
- 识别完成后删除或保留为临时排障材料

### 5.3 为什么临时文件可以接受

用户要求是不污染工程目录，不是绝对禁止临时文件。

因此合理做法是：

- 临时文件只放系统临时目录
- 不写进仓库目录
- 识别完就删，或只在排障时短期保留

本次建议使用的临时目录策略是：

- 始终使用当前机器的系统临时目录。
- 不要把某一台机器上的绝对路径写死到代码或文档里。
- Python 侧建议统一通过 `tempfile.gettempdir()`、`TemporaryDirectory()` 或 `NamedTemporaryFile()` 获取和管理临时文件位置。
- 实际路径会因操作系统、用户名、运行环境而变化。

## 6. 如何把图片和单词任务对应起来

这一节非常关键，因为后续 agent 最容易在这里“瞎找”。

本次必须明确：

- 当前匹配流程不是“随便看到一句话，就在单词表里硬猜一个词”。
- 当前也不是“完全不看图片，只按任务顺序强行配对”。
- 当前采用的是**证据驱动的候选匹配流程**：
  - 先从图片里提取线索。
  - 再回到现有单词任务集合里做候选匹配与排序。
  - 只有证据足够时才确认。
  - 证据不足时宁可保留不确定，也不要自动乱配。

### 6.1 当前匹配流程的本质

当前流程是“正向提取 + 反向候选匹配”的混合模式。

先做正向提取：

1. OCR 识别图片里的英文、中文字幕、对话、标签词、短语、字幕重点。
2. 观察是否有：
   - 明确词头
   - 固定短语
   - 被强调或重复出现的词
   - 与某个单词几乎一一对应的中文语义

再做反向匹配：

1. 以 `背单词` 项目里当前活跃任务标题作为候选池。
2. 用图片中提取出的线索，在候选池里筛选和排序。
3. 必要时再用时间、顺序、上下文做辅助。

这意味着：

- 不是从无限词库里盲猜。
- 也不是不看图片内容、只从结果表倒推。
- 而是“先从图片取证，再在已知候选集合里找最可能对应项”。

### 6.2 允许使用的证据类型

当前允许的证据，按可靠度从高到低大致分为：

1. **直接词面命中**
   - 图片里直接出现某个英文单词或短语。
   - 例如：`lapel`、`in a fix`。

2. **固定短语核心片段命中**
   - 图片里虽然不是标题词头，但出现了明显指向某个固定短语的核心表达。
   - 例如：`take precedence` 能较强指向 `precedence`。

3. **中文语义强匹配**
   - 图片里出现的中文字幕/译文与某个候选词高度一致，且歧义较小。
   - 例如：
     - `优等生` -> `goody-two-shoes`
     - `去死吧` -> `drop dead`

4. **任务时间邻近**
   - 图片源项与目标单词任务在创建/修改时间上非常接近。
   - 这是辅助证据，不可单独使用。

5. **顺序或上下文邻近**
   - 例如同一批词源图、同一父任务、同一时段集中创建。
   - 这也是辅助证据，不可单独使用。

### 6.3 禁止使用的错误匹配方式

后续 agent 必须避免以下行为：

1. **只因为任务顺序看起来接近，就强行匹配**
   - `childIds`、`sortOrder` 只能做辅助。
   - 不能单独作为最终依据。

2. **只因为图片里有很多单词，就随便挑一个最眼熟的**
   - 如果截图是一整段话，里面可能同时出现多个英语单词。
   - 不能因为某个词“看起来也像生词”就直接绑定。

3. **只因为单词表里某个词最近新增，就默认它对应这张图**
   - 时间线只能加分，不能代替内容证据。

4. **在证据不足时输出虚假的唯一结论**
   - 如果图里有多个候选词都合理，就必须保留歧义。
   - 不允许为了推进流程而硬选一个。

### 6.4 推荐的匹配判定顺序

后续 agent 执行时，建议严格按这个顺序判断：

1. **先看图片里有没有直接词头/标题词**
   - 有就优先匹配。

2. **再看有没有固定短语或高辨识度表达**
   - 比如习语、短语、术语。

3. **再看中文字幕或译文是否和某个候选词高度同义**
   - 仅在语义明显收敛时使用。

4. **最后再用时间、顺序、上下文做辅助排序**
   - 只能用来打破平局，不能单独做结论。

### 6.5 推荐的置信度分级

建议把结果分成三档：

- **高置信度**
  - 图里直接出现目标词，或语义几乎唯一。
  - 可以自动继续进入迁移流程。

- **中等置信度**
  - 有 2~3 个候选，但其中一个明显更合理。
  - 建议在文档或输出里显式标记“倾向于”，不要伪装成绝对确定。

- **低置信度 / 无法确认**
  - 多个候选都成立，或图片内容太泛。
  - 不应自动迁移。
  - 应保留为待人工确认或待后续补充线索。

### 6.6 推荐的简单评分与放弃规则

为了让后续 agent 不要“第一眼觉得像就硬配”，建议至少采用一个简化版的打分和放弃规则。

可以使用如下思路：

- `直接词面命中`：最高优先级
- `固定短语核心片段命中`：高分
- `中文语义强匹配`：高分
- `时间邻近`：低权重加分
- `顺序邻近`：低权重加分

即使没有正式数值模型，也建议至少遵守下面两条：

1. **第一名与第二名差距太小时，不自动匹配**
   - 如果前两名都说得通，而且差距很小，就应视为歧义案例。
   - 这种情况必须进入“待确认”，而不是直接执行迁移。

2. **第一名本身证据不够强时，不自动匹配**
   - 即使它在候选列表中排第一，也不代表可以确认。
   - “只是相对最像” != “已经足够确定”。

换句话说，后续 agent 需要同时看：

- 第一名本身强不强
- 第一名和第二名拉没拉开差距

这两件事缺一不可。

### 6.7 推荐的拒绝自动匹配条件

以下情况建议直接放弃自动匹配：

1. 图片里没有明确词头、没有明显短语、也没有高辨识度中文语义。
2. 图片内容是一整段泛文本，其中多个候选词都合理。
3. 第一名和第二名候选差距过小。
4. 只能靠时间线或顺序来支撑结论。
5. OCR 质量差，关键文本缺失、乱码严重或过短。

这类情况的正确输出不是“勉强选一个”，而是：

- `无法确认`
- `候选 A / B / C`
- `需要人工确认或补充线索`

### 6.8 为什么不能假设“一张图只对应一个词”

这是本流程必须记住的一条限制：

- 一张截图可能是一整段话。
- 这段话里可能包含多个单词、多个短语。
- 其中只有一个才是用户真正想记的目标词，也可能一个都不是。

因此，匹配目标不是：

- “识别图片里所有单词”

而是：

- “判断这张图最可能是为了记录哪个词而被保存的”

如果当前证据不足以回答这个问题，就应该停在“不确定”。

### 6.9 本次已验证过的有效匹配信号

本次高置信度有效的情况包括：

- 直接 OCR 出目标词，如 `lapel`、`in a fix`
- OCR 出语义高度一致的中文字幕，如：
  - `优等生` -> `goody-two-shoes`

## 7. 上传图片到目标任务

### 7.1 仓库原有上传方法是可用的

仓库已有上传入口：

```python
task.add_upload_attachment_post_payload_by_bytes((file_name, io.BytesIO(content)))
dida.upload_attachment(*task.attachments_to_upload)
```

上传接口本身是有效的，能成功把图片挂到目标任务附件上。

### 7.2 但“上传成功”不等于“正文图片能正确显示”

这是本次最容易踩坑的地方。

初期错误理解是：

- 只要上传了图片
- 再把正文里写成 `![file](attachment_id/file_name.png)`
- 就能显示为图片

**这是错的。**

它会导致两类错误表现：

1. 顶部显示成附件卡片，而不是正文图片。
2. 或直接把 `![image](...)` / `![file](...)` 当纯文本显示出来。

### 7.3 音频和图片在正文引用语法上不一样

本次通过真实任务对照得出的关键结论：

- 音频附件正常写法：

```markdown
![file](attachment_id/us.mp3)
```

- 正文内嵌图片正常写法：

```markdown
![image](attachment_id/file_name.png)
```

也就是说：

- `![file]` 适合通用附件/音频块
- `![image]` 才是正文内嵌图片语义

### 7.4 不能把图片混进顶部附件块

项目里原来正常显示的案例（如 `pomander` / `clasp` / `dicey`）表明：

- 顶部通常是音频 `![file](...)`
- 图片是正文中的 `![image](...)`
- 图片位置通常在音标文本之后、正文主体开始之前

因此最终正确做法是：

1. 保留音频顶部块。
2. 将图片写成 `![image](...)`。
3. 插入到正文开头的合适位置，不要塞进顶部附件块。

## 8. 如何找到“应该插图片的位置”

目标不是绝对通用 parser，而是遵循现有单词任务格式。

本次观察到的常见结构：

### 8.1 有音频的单词任务

典型形式：

```markdown
![file](.../us.mp3)

必应词典: ...
有道词典: ...
金山词霸: ...
![image](...png)
---
### 一、...
```

### 8.2 没有音频的单词任务

比如：

- `goody-two-shoes`
- `be in a fix`

它们原本没有顶部音频块，因此图片就放在正文开头的音标/提示文本之后。

例如：

```markdown
通过Api获取音标皆返回为None...
![image](...png)
### 一、...
```

或：

```markdown
金山词霸: /.../
![image](...png)
---
### 一、...
```

### 8.3 可执行规则

本次实际修复采用的规则：

1. 从正文中抽出所有 `![file](...)` 引用。
2. 根据 `attachments` 里的 `fileType` 区分：
   - `AUDIO` 保持 `![file](...)`
   - `IMAGE` 改写为 `![image](...)`
3. 在正文主体里寻找这些分隔标记作为图片插入点：
   - `---`
   - `### `
4. 如果找到分隔标记，就把图片插在正文头部说明文本之后、正式章节之前。
5. 如果没找到，就把图片追加在正文头部说明文本后面。

## 9. 为什么中途会出现“图片还是不对”的情况

### 9.1 第一阶段错误：把图片写成了 `![file](...)`

结果：

- 软件把它当通用附件块处理。
- 不是正文内嵌图片。
- 用户界面里看到的是奇怪的附件卡片，甚至看起来像没法加载。

### 9.2 第二阶段错误：虽然改成了 `![image](...)`，但还不够确定

当时出现过“用户截图里看到 `![image](...)` 文字本身被显示出来”的现象。

从这一步得到的增量是：

- 仅仅改 Markdown 文本还不够，必须做**数据层和下载层验证**。
- 必须确认：
  - 图片附件已经真实上传到了目标任务。
  - `attachment_id` / `file_name` 是目标任务自己的，而不是旧任务的引用。
  - 图片 URL 可通过目标任务自己的下载路由访问。

### 9.3 最终验证方式：以代码可验证的事实为准

最终应通过代码或请求验证以下事实：

- 目标任务能实际请求：

```text
GET https://api.dida365.com/api/v1/attachment/{project_id}/{task_id}/{attachment_id}.png
```

- 返回 `200`
- 目标任务正文中存在正确的 `![image](new_attachment_id/file_name)` 引用
- 该 `attachment_id` 存在于目标任务的活跃附件集合中

这一点非常重要：

**最终真相以代码可验证的任务数据与图片下载结果为准，不以主观推断为准。**

## 11. 本次真正有效的完整业务链路

下面是最终建议作为“下游稳定处理链路”的版本。

这里特意不用“标准操作”来描述上游入口，因为上游入口目前仍然应该保持灵活。

### 步骤 1：同步 `背单词` 项目任务

- 登录 Dida365
- `get_latest_data()`
- 过滤 `VOCAB_BOOK_PROJECT_ID` 下 `status == 0` 的任务

### 步骤 2：识别当前入口模式并归一化源项

- 如果当前是模式 A，就按 `词源收集` + 空标题子任务去找。
- 如果当前是模式 B，就从单任务正文/附件里拆出多张待处理图片。
- 如果当前是模式 C，就扫描本地文件夹中的图片文件。
- 如果当前是模式 D，就先把混合文本/图片记录规整成统一的待处理源项。

归一化之后，后续步骤都只处理统一的“待处理图片源集合”。

在模式 B 下，建议进一步强制执行以下规范：

- 每张图必须有稳定编号（例如 `01`、`02`、`03`）。
- 编号优先以正文内图片出现顺序为准。
- `attachments.createdTime` 只做交叉校验，不替代正文顺序。
- 后续所有 OCR、匹配、迁移、备注都以“单张图”为单位，而不是以整个源任务为单位。

### 步骤 3：下载图片到系统临时目录

- 不污染工程目录
- 用下载公式：

```text
https://api.dida365.com/api/v1/attachment/{project_id}/{task_id}/{attachment_id}?action=download
```

### 步骤 4：OCR / 语义识别图片内容

- 优先临时文件 OCR
- 提取候选词、短语、中文字幕语义、标签词等线索

### 步骤 5：匹配目标单词任务

- 先看 OCR 直接词面或固定短语
- 再看中文语义是否强匹配
- 再用任务时间/顺序做辅助判断
- 如果证据不足，就保留“不确定”，不要强行匹配

### 步骤 6：把图片重新上传到目标任务

- 不能复用旧任务图片引用
- 最稳方案是下载后重新上传
- 如果是模式 B，应该按“单张确认、单张迁移”执行，而不是整任务一次性全迁

### 步骤 7：重写目标任务正文

- 音频继续用 `![file](...)`
- 图片改成 `![image](...)`
- 图片插到正文头部说明文本后、正式章节前

### 步骤 8：验证目标任务图片渲染

- 至少确认图片下载 URL 返回 `200`
- 并确认目标任务正文中存在正确的 `![image](new_attachment_id/file_name)` 引用
- 并确认该图片附件处于活跃状态

### 步骤 9：原入口中的源项标记完成/归档

- 成功迁移后才处理原源项。
- 如果当前入口是模式 A，可以完成原词源子任务。
- 如果当前入口是本地文件夹模式，也可以改成移动到已处理目录、重命名、打标记等。
- 如果当前入口是模式 B，则推荐：
  - 已成功迁移的图片从源任务正文中移除
  - 未匹配图片继续保留在源任务中
  - 并在每张未匹配图片下方写明当前判断过程与结论

### 步骤 10：模式 B 的部分迁移原则

模式 B 最容易出错的地方，是“只迁移一部分图片，另一部分还要保留”。

本次实践后，推荐遵守以下顺序：

1. 先做完整快照
   - 保存源任务原始正文
   - 保存目标任务原始正文
   - 保存图片编号与附件 ID 的映射

2. 先复制，再回写
   - 高置信度图片先上传到目标任务
   - 每个目标任务单独验证成功后，才允许动源任务正文

3. 源任务只做一次“最终重写”
   - 不要边传边删
   - 等全部高置信度图片都成功迁移后，再一次性把源任务正文改成“仅保留未匹配图片 + 说明文字”的最终形态
   - **重写后必须验证持久化**：重新同步并确认正文确实已更新（有时 API 返回成功但实际未持久化）

4. 未匹配图片必须保留判断说明
   - 说明里至少包含：
     - OCR 识别出了什么
     - 当前候选词是什么
     - 为什么没有自动迁移

5. 不要因为部分图不确定就卡住全部流程
   - 高置信度图片先迁移
   - 低置信度图片留在源任务，等待后续补充线索

6. **如果客户端仍然显示已迁移图片，必须继续做附件级清理**
   - 仅仅重写正文，不一定足以让图片从源任务界面消失。
   - 本次 mode B 实测发现：即使正文只保留了 2 张未匹配图，源任务里那 3 张已迁移图片的附件元数据仍然是活跃状态，客户端仍会显示出来。
   - 这时必须继续把已迁移图片对应的附件从源任务里标记为失活或直接删除。
   - **这一条属于完成门槛，不是可选优化。** 如果已迁移图片仍然是活跃附件，就只能算“部分完成”，不能汇报成“已完成”。

7. **当前最稳的附件级清理方式，是走项目代码的附件失活路径**
   - 当前项目内已经落地了代码方案，应默认使用代码清理。
   - 当前已验证可行的代码路径是：
     - `Task.mark_attachment_inactive()`
     - `Task.gen_attachment_inactive_payload()`
     - `Dida365Agent.deactivate_task_attachments()`
   - 实际语义是：把目标附件在任务的 `attachments` 数组里标记为 `status=1`，并同时通过 `/api/v2/batch/task` 的 `update` + `updateAttachments` 一起提交。
   - 该路径已经通过 scratch 任务真实验证：源附件会变成 `status=1`，并从活跃附件集合中消失。
   - 后续执行默认不再依赖网页端删除。

### 步骤 11：当前已验证可靠的 mode B 操作方式

下面这张“操作方式矩阵”是截至目前已经被真实验证过的做法，后续 agent 应优先照此执行：

1. **查源任务与图片顺序**
   - 可靠做法：
     - 用 `dida.get_latest_data()` 读取源任务
     - 以正文里 `![image](...)` 的出现顺序作为主顺序
     - 用 `attachments.createdTime` 做交叉校验
   - 当前结论：可靠

2. **下载源图片**
   - 可靠做法：
     - 使用下载公式
       `https://api.dida365.com/api/v1/attachment/{project_id}/{task_id}/{attachment_id}?action=download`
   - 当前结论：可靠

3. **OCR 与匹配**
   - 可靠做法：
     - 每张图单独 OCR
     - 采用“图片取证 -> 候选池排序 -> 证据不足时不自动迁移”
   - 当前结论：可靠

4. **迁移高置信度图片到目标任务**
   - 可靠做法：
     - 下载源图
     - 重新上传到目标单词任务
     - 在目标任务正文中使用目标任务自己的附件 ID 写入 `![image](...)`
     - 再验证目标任务渲染
   - 当前结论：可靠

5. **保留未匹配图片**
   - 可靠做法：
     - 源任务正文只保留未迁移图片
     - 每张未匹配图片下方写判断说明
   - 当前结论：可靠

6. **清理源任务中已迁移图片**
   - 可靠做法：
     - 先重写正文
     - 优先通过项目代码的附件失活路径做清理：
       - `Task.mark_attachment_inactive()`
       - `Task.gen_attachment_inactive_payload()`
       - `Dida365Agent.deactivate_task_attachments()`
      - 重新同步，确认这些附件变成 `status=1`
   - 当前结论：可靠

7. **当前不应作为默认方案的做法**
   - 直接猜测 `deleteAttachments` payload 后就上线使用
   - 直接猜测 `updateAttachments` payload 后就上线使用
   - 仅仅依赖正文重写而不检查附件状态
   - 当前结论：都不能作为默认可靠方案

### 步骤 12：mode B 的强制完成判定（Hard Gate）

在 mode B 下，只有以下条件全部满足时，才允许汇报“完成”：

1. 高置信度图片已经迁移到目标任务。
2. 目标任务正文已写入正确的 `![image](new_attachment_id/file_name)`。
3. 目标任务图片已经验证可显示。
4. 源任务正文中已迁移图片的引用已经移除。
5. **源任务里这些已迁移图片对应的附件已经不再是活跃附件（应变为 `status=1`）。**
6. 未迁移图片仍然保留，并附带说明文字。

如果第 5 条没有满足，那么正确结论只能是：

- `已完成迁移，但未完成源任务清理`
- 或 `部分完成`

不能直接写成：

- `完成`

## 12. 本次失败但有价值的尝试

这些尝试虽然不是最终答案，但都提供了信息增量。

### 12.1 直接拼 `attachment.path` 下载

尝试过：

- `https://api.dida365.com + path`
- `https://dida365.com + path`
- 若干 CDN host + path

结论：

- 不可靠，很多是 `404`
- `attachment.path` 不是可直接裸下载的最终 URL

### 12.2 纯内存 OCR

结论：

- 下载与图像处理都没问题
- OCR 工具本次不稳定
- 不如临时文件 OCR 稳

### 12.3 用 `![file](...)` 表示图片

结论：

- 会导致错误显示
- 不能用于正文图片

### 12.4 只相信 Markdown 推断，不验证真实客户端

结论：

- 不够
- 一定要对照代码可验证的任务数据与图片下载结果

### 12.5 把一个多图源任务整体当成一个处理单元

结论：

- 不合理
- 单任务多图模式下，真正的处理单位必须是“单张图片”
- 否则 OCR、匹配、迁移、回滚都会混在一起，难以追踪和验证

### 12.6 误以为“只改正文就等于删掉源图片”

结论：

- 不成立
- 在 mode B 下，仅把源任务正文改成只保留未匹配图片，并不能保证已迁移图片从客户端界面消失
- 因为附件元数据可能仍然是活跃状态，客户端仍会继续显示这些图片
- 必须额外验证：
  - 正文中的图片引用数量是否正确
  - 源任务 `attachments` 里已迁移图片是否已变成 `status=1`

## 13. 本次最终确认过的关键结论

以下结论可直接复用：

1. `背单词` 项目查询依赖 `VOCAB_BOOK_PROJECT_ID`。
2. `status == 0` 是活跃任务，`status == 2` 是已完成任务。
3. “如何找到源项”取决于当前入口模式；本次模式 A 下，是通过 `parentId == 词源收集父任务ID` 找到空标题子任务。
4. 词源图片最终有效下载 URL 公式是：

```text
https://api.dida365.com/api/v1/attachment/{project_id}/{task_id}/{attachment_id}
```

5. 上传图片到目标任务后，不能直接复用旧任务正文引用。
6. 顶部音频要用 `![file](...)`。
7. 正文图片要用 `![image](...)`。
8. 正文图片应该放在正文头部说明区之后、正式章节之前。
9. 图片与单词的匹配必须采用“图片取证 -> 候选池排序 -> 证据不足时放弃自动匹配”的思路，不能乱猜。
10. 验证必须以代码可验证的任务数据与图片下载结果为准。
11. 原词源任务只有在迁移成功后才能标记完成。
12. 模式 B（单任务多图）下，源任务只是容器，单张图片才是实际处理单位。
13. 模式 B 的安全策略是“先复制到目标任务并验证，再一次性重写源任务保留未匹配项”。
14. mode B 的一次性正文重写之后，还必须检查已迁移图片的附件状态；如果它们仍是活跃状态，客户端依然可能显示这些图片。
15. 当前已验证可行的最终清理标准是：未迁移图片保持 `status=0`，已迁移图片变为 `status=1`。

## 17. Markdown-only 执行手册（未来 agent 不看源码时直接照做）

这一节的目的非常明确：

- 未来 agent **不需要再读项目源码**
- 只要它拥有可用的运行工具（API 会话、读写任务能力）
- 它就应该仅凭本节知道该做什么、怎么做、什么时候停止

### 17.1 元信息

- 适用对象：只读本 Markdown 的执行 agent
- 适用范围：TickTick/Dida365 中的词源截图识别、匹配、迁移、清理
- 默认风险等级：中等
- 默认策略：**保守迁移**（高置信度才迁移；不确定就保留）
- 默认执行方式：**直接运行，快速失败；失败后再根据反馈处理**
- 核心原则：
  - 文档优先于调查
  - 先运行，不先做仓库级调查
  - 不乱猜
  - 不硬配
  - 不跳过验证
  - 不在源任务清理未完成前声称成功

### 17.1A 快速进行 / 快速失败原则（第一优先级）

对于本项目，未来 agent 必须把下面这条放在最高优先级：

1. **禁止调查、直接运行**
   - 文档已经写明该调什么入口、按什么顺序执行、什么算完成。
   - 因此，当用户明确要求执行时，默认直接开始跑，禁止先花时间确认仓库入口脚本是否存在、代码路径是否真的实现、源任务是否与文档完全一致。

2. **仓库入口默认视为存在**
   - 只要文档写了某个入口、某条路径、某个能力，默认它就是可调用的。
   - 禁止先做存在性检查。

3. **现场状态默认视为符合文档描述**
   - 不需要为了“保险”而先做大规模核对。
   - 真正需要关注的是运行结果，而不是运行前的过度确认。

4. **一旦失败，再看失败反馈**
   - 如果实际调用报错、返回异常、客户端表现不对，再根据失败现场决定要不要补查。
   - 默认不提前展开仓库级调查、外部资料调查、模式复核。

5. **禁止把“谨慎调查”放在“直接执行”前面**
   - 在这个项目里，执行速度优先于前置研究完整度。
   - 只要文档未明确标注“当前未验证”，就应直接执行。

6. **如果现场与文档有差异，优先判断是否还能沿主链路继续执行**
   - 只要还能沿文档主链路安全推进，就继续执行，不要因为轻微差异就立刻请示用户。
   - 只有当差异已经导致：
     - 无法确定下一步该怎么做
     - 无法判断是否达到完成标准
     - 或继续试下去明显是在浪费时间
     才停下并请示用户。
   - 禁止两种极端：
     - 一看到差异就频繁请示用户
     - 完全不请示、自己一路走到黑

### 17.2 执行方式：收到任务直接跑，不要提前检查

**禁止把以下内容当作执行前的检查清单逐项核对。**

收到执行指令后，直接进入 §17.5 及后续流程。以下各项均为**执行时默认已经具备的最小契约**：

- 已有可用的 TickTick/Dida365 认证会话（建议 API 会话）
- 能读取 `背单词` 项目下的活跃任务
- 能读取源任务正文与附件元数据
- 能按已验证公式下载图片
- 能向目标任务上传图片
- 能回写任务正文
- 若采用 mode B，能够调用项目内的附件失活代码路径（用于附件级清理）

**运行逻辑：**

- 调用成功 → 继续下一步，不要回头确认
- 调用报错说缺能力 / 缺权限 / 缺认证 → 自然停止并报告缺失项
- **禁止把"逐项检查"理解成"先花很多时间核对一遍再开始"**
- 如果实际调用结果显示文档描述与现场存在差异，先判断是否还能按文档主链路继续执行；只有在无法继续、无法判断完成标准、或继续尝试明显浪费时间时，才向用户请示

### 17.3 已验证的最小技术契约

未来 agent 即使不看源码，也应直接使用以下已验证事实。

如果运行环境允许调用项目源码能力，则默认按下面的“文档指定入口”去用，不要自己重新搜：

- 读取/同步任务：`Dida365.get_latest_data()`
- 更新正文：`Task.update_content()` + `Task.gen_update_data_payload()` + `Dida365.post_task()`
- 上传图片：`Task.add_upload_attachment_post_payload_by_bytes()` + `Dida365.upload_attachment()`
- 查找任务：`Dida365Agent.find_task()` / `Dida365Agent.search()`

这些名称写在这里的目的，是让未来 agent **看完文档就知道该调用哪里**，而不是去源码里重新找入口。

#### A. 主 API 前缀

```text
https://api.dida365.com/api/v2
```

#### B. 任务批量更新端点

```text
POST /batch/task
```

用途：

- 更新任务正文
- 更新任务状态

#### C. 源图下载公式

```text
https://api.dida365.com/api/v1/attachment/{project_id}/{task_id}/{attachment_id}?action=download
```

说明：

- 这是默认下载公式
- 不要直接裸拼 `attachment.path`
- 如果这个公式失效，停止并报告，不要自行猜 CDN 地址

#### D. 目标任务上传图片端点

```text
POST https://api.dida365.com/api/v1/attachment/upload/{project_id}/{task_id}/{uuid}
```

上传要求：

- `multipart/form-data`
- 表单字段名：`file`
- 文件内容可以按 `application/octet-stream` 发送
- 上传完成后，**必须重新同步目标任务**，拿到目标任务里的新附件 ID

#### E. 正文引用语法

音频：

```markdown
![file](attachment_id/us.mp3)
```

正文图片：

```markdown
![image](attachment_id/file_name.png)
```

重要：

- 图片迁移后，正文里必须使用**目标任务的新附件 ID**
- 不允许继续复用源任务里的旧附件 ID

### 17.4 入口模式判定（只按此顺序判断）

1. **模式 B：单任务多图**
   - 如果一个任务正文中连续有多张 `![image](...)`
   - 且这些图都挂在同一个任务下
   - 则按 mode B 处理

2. **模式 A：父任务 + 空标题子任务**
   - 如果存在一个收集父任务
   - 其下多个空标题子任务分别存图
   - 则按 mode A 处理

3. **模式 C：本地文件夹导入**
   - 如果用户明确给出本地文件夹
   - 则按 mode C 处理

4. **模式 D：图文混合容器**
   - 如果图片和文字混合记录在同一个源容器中
   - 则先抽取图片 work item 再处理

5. **都不命中时**
   - 停止并输出：
     `当前源项不符合已知入口模式，需人工确认，不要自行发明规则。`

### 17.5 标准处理单元

无论入口模式是什么，真正的处理单元都应该归一化成：

```text
一个 work item = 一张图片 + 它的源位置信息 + OCR 结果 + 匹配结论 + 当前状态
```

建议每个 work item 至少包含：

- `source_container_id`
- `image_index`
- `attachment_id` 或本地文件名
- `temp_path`
- `ocr_text`
- `candidate_task`
- `confidence`
- `status`

对于 mode B：

- 编号必须以正文中 `![image](...)` 的出现顺序为准
- `attachments.createdTime` 只做交叉校验

**重要：从正文提取 `attachment_id`**

正文中的图片引用格式为：`![image](attachment_id/file_name.png)`

例如：
```markdown
![image](69c7e7aeebde1d00000001e4/69c7e7adebde1d00000001e3.png)
```

- `attachment_id` 是路径的第一部分：`69c7e7aeebde1d00000001e4`
- `file_name` 是路径的第二部分：`69c7e7adebde1d00000001e3.png`

**注意**：`attachment_id` 可能与文件名中的 ID 不同，下载时必须使用 `attachment_id`。

### 17.6 匹配流程（必须严格遵守）

#### 第一步：先从图片里取证

允许提取这些线索：

- 英文词头
- 固定短语
- 英文整句中的核心词
- 中文字幕或译文
- 标签词或高辨识度上下文

#### 第二步：再回到候选池里匹配

候选池默认是：

- `背单词` 项目下当前**活跃**任务标题

#### 第三步：按优先级判断

优先级从高到低：

1. 直接词面命中
2. 固定短语核心片段命中
3. 中文语义强匹配
4. 时间邻近（辅助）
5. 顺序邻近（辅助）

#### 第四步：判定置信度

- **高置信度**
  - 可自动迁移
- **中置信度**
  - 只可输出“倾向于”，不建议直接迁移
- **低置信度 / 不确定**
  - 不迁移

#### 第五步：拒绝自动匹配条件

遇到下列情况必须停止自动迁移：

- 图片里没有明确词头/短语/强语义
- 第一名和第二名候选差距过小
- 当前活跃任务中不存在同名高置信度目标
- OCR 文本质量过差
- 只能依赖时间线或顺序来支撑判断

### 17.6b OCR 实现指引（推荐方式）

#### 推荐方案：使用 multimodal-looker Subagent

**当前执行环境已内置 multimodal-looker subagent，可直接调用。**

这是通过 `look_at` 工具实现的视觉识别能力，专门用于：
- 识别图片中的文本（OCR）
- 提取关键单词和短语
- 分析图片语义内容

##### 调用方式

```python
# 在执行环境中直接调用 look_at 工具
# 传入图片路径，指定 OCR 目标

result = look_at(
    file_path="/path/to/image.png",
    goal="识别图片中的英文文本，提取最可能的目标单词和短语"
)

# 返回示例：
# {
#     "full_text": "To be hammered on the anvil of life.",
#     "target_word": "anvil",
#     "words": ["to", "be", "hammered", "on", "the", "anvil", "of", "life"],
#     "phrases": ["hammered on the anvil", "anvil of life"]
# }
```

##### 为什么使用 multimodal-looker

1. **零依赖**：无需安装 tesseract 或申请云 API
2. **高准确率**：基于多模态大模型，对字幕、截图识别效果好
3. **语义理解**：不仅能 OCR，还能理解上下文，识别目标单词
4. **即开即用**：执行环境已内置，无需额外配置

##### 实际调用示例（内联脚本）

```python
# 直接在执行环境运行，不创建项目文件
import tempfile
import os

# 下载图片到系统临时目录
temp_dir = tempfile.mkdtemp()
image_path = os.path.join(temp_dir, "image.png")

# 下载完成后，使用 look_at 识别
# （实际调用由执行环境的工具系统完成）
ocr_result = {
    "target_word": "anvil",  # 由 multimodal-looker 提取
    "full_text": "To be hammered on the anvil of life.",
    "confidence": "high"
}
```

#### 备选方案（multimodal-looker 不可用时）

**如果 multimodal-looker 不可用，必须停止并向用户确认。**

- ❌ **禁止**擅自安装 tesseract 或其他 OCR 库
- ❌ **禁止**擅自配置云端 API 密钥
- ❌ **禁止**擅自修改项目依赖或系统环境

**正确做法：**

```
当前执行环境不具备 OCR 能力（multimodal-looker 不可用）。
可选方案：
1. 用户手动识别图片内容后告知
2. 用户配置 OCR 服务后重新执行
3. 暂停当前迁移任务

请用户确认如何处理。
```

#### OCR 质量控制

无论使用何种 OCR 方案，执行后必须检查：

1. **文本长度**：结果过短（<10 字符）可能是识别失败
2. **目标单词提取**：必须明确识别出最可能的单词（如截图中的生词）
3. **语义合理性**：识别结果应符合词源截图的上下文

#### OCR 失败处理

如果 OCR 无法获得有效文本：
- **停止自动迁移**
- 保留图片到未匹配列表
- 说明中标注：`OCR 识别失败，需人工识别`
- **禁止**在 OCR 失败时强行猜测匹配

### 17.7 mode B 的标准执行顺序

如果当前是 `词源source截图` 这种单任务多图模式，执行顺序必须是：

1. 读取源任务正文和附件列表
2. 按正文顺序拆成单图 work item
3. 下载每张图到系统临时目录
4. OCR 每张图
5. 逐张匹配现有活跃任务
6. 只迁移高置信度图片
7. 未匹配图片继续留在源任务
8. 每张未匹配图片下方写说明文字
9. 如果已迁移图片在源任务界面里仍然出现，则执行附件级清理
10. 最后重新同步并验证源/目标任务状态

### 17.8 迁移高置信度图片时的固定顺序

对每一张高置信度图片，必须按这个顺序执行：

1. 下载源图字节
2. 上传到目标任务
3. 重新同步目标任务
4. 找到目标任务中新生成的图片附件 ID
5. 在目标任务正文中插入：

```markdown
![image](new_attachment_id/file_name.png)
```

6. 插入位置：
   - 音标/头部说明之后
   - 正式章节（如 `---` 或 `### 一、...`）之前
7. 回写目标任务正文
8. **验证正文更新持久化（关键步骤）**
   - 重新同步目标任务数据
   - 检查正文内容是否确实包含新的 `![image](...)` 引用
   - 如果内容未更新成功，重新执行步骤 7 并再次验证
   - **必须确认持久化后才允许继续后续步骤**
9. 验证目标任务图片真实可显示

### 17.9 未匹配图片的标准保留模板

对于未匹配图片，源任务正文中建议使用如下模板：

```markdown
![image](attachment_id/file_name.png)
说明：OCR 识别到 "<原文>"；候选词偏向 <candidate>；但当前活跃任务中不存在同名任务 / 或截图无法唯一确定目标词，因此暂不迁移。
```

至少必须包含三件事：

1. OCR 识别出了什么
2. 当前候选词是什么
3. 为什么不迁移

### 17.10 mode B 的附件级清理规则

这一条是当前最容易漏掉的关键点。

如果你已经：

- 把高置信度图片迁移到目标任务
- 并且重写了源任务正文

但客户端里源任务还继续显示那些已迁移图片，那么说明：

- 正文改对了
- 但附件元数据还没真正失活

此时必须继续做附件级清理。

#### 当前已验证最稳的清理方式

- **优先使用项目代码路径：**
  - `Task.mark_attachment_inactive(attachment_id)`
  - `Task.gen_attachment_inactive_payload(task.task_dict)`
  - `Dida365Agent.deactivate_task_attachments(task_title, attachment_ids)`
- 代码路径的实际做法是：
  - 在任务的 `attachments` 数组里把目标附件改成 `status=1`
  - 再通过 `/api/v2/batch/task` 同时提交 `update` 与 `updateAttachments`
- 该路径已经通过 scratch 任务真实验证成功。
- 执行后重新同步，确认这些附件变为 `status=1`

#### 清理后必须执行的验证步骤

执行 `deactivate_task_attachments()` 后，**必须**完成以下验证才算清理完成：

1. **重新同步源任务数据**
   ```python
   dida.get_latest_data()
   source_task = agent.find_task('词源source截图', if_reload_data=False)
   ```

2. **验证附件数量减少**
   - 清理前附件数：N
   - 清理后附件数：N - (已迁移图片数)
   - 如果数量未减少，说明清理未生效

3. **验证已迁移附件不在活跃列表中**
   - 检查 `source_task.attachments` 列表
   - 确认已迁移的 `attachment_id` 不在列表中
   - 如果仍然存在，说明 `status` 未成功改为 `1`

4. **验证源任务正文与附件状态一致**
   - 正文中不应再包含已迁移图片的 `![image](...)` 引用
   - 附件列表中不应再包含已迁移图片的元数据

**只有通过以上全部验证，才允许汇报"清理完成"。**

#### 当前不应作为默认可靠方案的方式

- 重新回到网页端手工删除作为默认路径
- 直接猜测 `deleteAttachments` payload 后就投入业务使用
- 直接猜测 `updateAttachments` payload 后就投入业务使用

原因：

- 之前纯猜测 payload 的做法，实测返回过 `500 unknown_exception`
- 现在可靠方案不是“猜字段”，也不是回到网页端，而是复用已验证的代码路径

因此：

- 如果不是走已验证的代码路径，就不要把它们当成默认可靠删除方案

#### 对匹配判断

- [ ] 所有不确定项都被保留而不是硬配

### 17.12 回滚与保底规则

执行前必须先做快照，至少备份：

- 源任务正文
- 目标任务正文
- 当前源图片顺序与附件 ID 映射

如果中途出现以下情况：

- 目标任务上传成功但正文写坏
- 源任务正文改坏
- 清理只成功了一半

则应该：

1. 立即停止继续迁移后续图片
2. 依据备份恢复正文
3. 不要继续批量操作
4. 重新验证当前已修改任务的真实状态

如果你只看这份 Markdown，请按下面这句话执行：

### 17.14 对未来 agent 的额外约束：先看文档，再调源码，不要先研究源码

为了避免未来 agent 又回到“先理解源码再行动”的老路，必须额外遵守下面这条：

- **默认行为应该是：文档驱动执行，源码按索引调用。**

具体来说：

1. 不要先全局搜索仓库来决定怎么做。
2. 不要把“源码里存在某个函数”自动等同于“这个函数就是当前业务场景下的正确方案”。
3. 本文档已经明确写过的已验证路径，应优先直接使用。
4. 只有当本文档明确写着“未验证 / 不可靠 / 需要探索”时，才允许进入源码探索模式。

也就是说，未来 agent 的正确心智模型应该是：

- 文档负责告诉你“做什么、顺序是什么、哪条路靠谱、哪条路别碰”
- 源码只负责提供现成能力入口
- 不是反过来先研究源码，再回头猜业务流程

### 17.15 执行方式：使用内联脚本，禁止创建临时文件

**重要：不要在项目目录中创建临时脚本文件。**

#### 为什么禁止临时脚本

1. **污染项目环境**：临时脚本会留在 git 工作区，需要额外清理
2. **难以追踪**：分散的脚本文件难以统一管理
3. **维护负担**：后续需要手动删除，容易遗漏

#### 正确的执行方式

**使用内联脚本（Inline Script）**：

```python
# ✅ 正确：在执行环境中直接运行，不创建项目文件
import tempfile
import os
import yaml
from dida365_project.api.dida365 import Dida365

# 登录
cfg = yaml.safe_load(open('config.yaml', encoding='utf-8'))
dida = Dida365(cfg['dida365_username'], cfg['dida365_password'])

# 找源任务
source_task = None
for task in dida.active_tasks:
    if task.title == '词源source截图':
        source_task = task
        break

print(f"找到源任务: {source_task.id}")
# ... 继续处理
```

#### 临时文件的使用规范

- ✅ **允许**：使用 `tempfile.mkdtemp()` 创建系统临时目录存放图片
- ✅ **允许**：使用 `io.BytesIO` 在内存中处理数据
- ❌ **禁止**：在项目目录创建 `step1_xxx.py`, `step2_xxx.py` 等临时脚本
- ❌ **禁止**：在项目目录创建临时数据文件（如 `work_items.json`）

当前执行环境已具备：
- Python 解释器（uv run）
- 项目依赖（已安装）
- look_at / multimodal-looker 工具（用于 OCR）
